  Three tensions I keep tripping over

  1. Self vs external is decided at the breaking action, not at the read.
  The spec says validity is evaluated lazily ‚ÄĒ but the cause (case 2 vs case 3)
  can't be reconstructed at observation time unless you tagged it earlier.
  Self-invalidation requires "the reviewer's own action invalidated their own
  other queued change," and by the time some later read notices Y is invalid,
  the action that broke it may be many state mutations in the past. So:

  - After any reviewer action that mutates state ‚Üí walk that same reviewer's
  other queued changes; the ones that no longer validate are case 2 ‚Üí
  auto-dismiss now.
  - After any other state-mutating action ‚Üí do nothing eagerly. Later, when any
  reviewer reads/acts and finds an invalid queued change with no case-2 tag ‚Üí it
   must be case¬†3.

    "Lazy" then applies to when validity is recomputed, not to how cause is 
  determined. Worth pinning this down in the proposal.

  2. The cascade frontier isn't knowable up front.
  "Maximal recursive set the reviewer can accept" grows as acceptances promote
  dependents (¬ß11.3). If you accept root X, X's children may promote into the
  same reviewer's queue ‚ÄĒ and those get folded in. Bounded by other-owner
  routing and leaves. The natural shape:

    worklist = [root]
    applied  = []
    snapshot live state
    while worklist:
      c = pop in payload-topological order
      validate c against running state         ‚ĒÄ‚Ēź
      if invalid: rollback applied; return reason ‚Ēā  evolving-state
      apply c                                     ‚Ēā  validation
      promote c's dependents                      ‚Ēā  (the ¬ß12.2 note)
      add same-reviewer promotions to worklist   ‚ĒÄ‚Ēė
    commit

  The "non-equivalence" note in ¬ß12.2 is essentially declaring that this isn't
  the same as "validate the whole set against pre-cascade state then apply" ‚ÄĒ
  and shouldn't be re-derived as that.

  3. The Case-1 cascade is now bidirectional in cause.
  Phase 6 wired reject‚Üídescendants invalid. Phase 7 adds ancestor becomes 
  invalid ‚Üí descendants invalid. But ancestors become invalid via cases 1/2/3 ‚ÄĒ
  so case-1 can chain off case-2 or case-3 invalidations elsewhere. The
  invalidation propagator becomes a graph operation, not a single-step thing.
  Worth a precise recursive definition in the spec.

  This collapses my three-box diagram from before: cases 2 and 3 differ only by
  "is the actor the queue's own reviewer, acting in review capacity?" The walk
  itself is one walk over all queued changes.

  Implementation consequence worth flagging in the proposal: there's no 
  proposal-scoped shortcut. You either walk all queued changes after every
  state-mutating action, or you maintain a reverse-index from state-elements to
  queued-changes-that-depend-on-them. PRD ¬ß11.1 ("writes are serialized
  server-side") makes the naive global walk tractable ‚ÄĒ it's O(queued changes)
  per write, no concurrency. Probably fine for an initial implementation, with
  the indexed version as a later optimization if needed.

  This also strengthens the direct-action ‚Üí only case 3 observation: when owner
  U does a direct edit, U isn't acting as anyone's reviewer in that moment, so
  by the rule above no queued change can be case 2 against U's direct action ‚ÄĒ
  even if U has their own queued changes. Direct actions only ever produce case
  3.

  Good catch ‚ÄĒ the cleaner one-rule formulation only emerges once you stop

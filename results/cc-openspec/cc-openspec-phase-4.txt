  Threads I'd pull on

  A few things stand out as worth resolving before the proposal calcifies them:

  1. Where does the write lock live, and what's its surface?
  Middleware over mutating handlers is the cleanest now, but Phases 5‚Äď7 add
  review actions that mutate state without being the same HTTP shape (POST 
  /changes/{id}/accept, etc.). A reusable "write-section" primitive at the
  application layer (a context manager / wrapper that each write op invokes)
  survives both phases without re-plumbing. Worth deciding now so the spec
  captures the right abstraction.

  2. The "shared" check in ¬ß6.3 precondition 1 ‚ÄĒ what does it mean 
  operationally?
  PRD wording: "no taxon in the deletion region is reachable from more than one
  root." Because ¬ß3.3 already forbids in-tree diamonds, a U-owned taxon with
  multiple parents is necessarily in multiple root-trees. So "shared" reduces to
   "has more than one parent" ‚ÄĒ which is much cheaper to check than walking up
  to roots. Worth confirming this equivalence in the proposal so the
  implementation can use the parent-count shortcut without drifting from the
  spec wording. (And the halt-frontier other-owned children are explicitly
  outside the region, so their parent-count is irrelevant ‚ÄĒ the check only runs
  on the region.)

  3. Region computation ‚ÄĒ one pass or two?
  The natural shape is one traversal from N that:
  - adds U-owned nodes to the deletion set, recursing into their children,
  - on hitting an other-owned child, records the (parent‚Üíchild) edge for
  detachment and stops.

  Then validate preconditions on the assembled set, then apply. Two open
  questions:
  - Do we validate as we go (fail fast) or after assembly? Assembling first
  gives better error messages ("blocked by these shared taxa: ‚Ä¶").
  - What about a U-owned taxon reachable from N via two wholly-U paths within
  the region? It still only appears once in the deletion set (it's a node, not
  an edge), but it has 2 parents inside the region ‚Üí not shared with anything
  outside, fine. The "shared" check is about parents outside the region, not
  just parent-count ‚Č•¬†2. So the parent-count shortcut from thread #2 needs
  refining: "has a parent that isn't in the deletion region."

  That last point feels like the kind of thing where the spec wording and the
  natural implementation can drift, and it's worth nailing precisely in the
  change proposal.

  4. Edge-add (PUT /taxa/{parentId}/children/{childId}) brings the child's whole
   live subtree implicitly.
  The invariant module needs to validate across the resulting tree(s), which
  includes the child's subtree merging in. Want to make sure the
  invariant-engine API from Phase 3 actually supports "what if we added this
  edge" without committing ‚ÄĒ i.e. it can take a hypothetical edge and check. If
  it only validates the live state, we have a gap to close.

  5. Edit owner-reassignment.
  Reads owner from request, validates it's a registered non-null username (Phase
   2 user-registry lookup), reassigns. Subtleties:
  - A self reassignment (assigning to oneself) ‚ÄĒ allowed? PRD doesn't forbid it.
   Probably fine as a no-op-ish success.
  - Reassigning combined with rename in the same PATCH ‚ÄĒ atomic, both validated,
   both apply or neither.
  - After reassignment, the previous owner loses delete/edit rights instantly.
  Worth a test.

  6. State reset.
  Phase 4 doesn't add a new kind of state ‚ÄĒ taxa already exist from Phase 3's
  domain work. But it adds operations that create taxa from API input, so POST 
  /reset needs to clear them whether they came from fixtures or from POST /taxa.
   Probably already true, but worth confirming the reset spec doesn't need an
  addition.

‚Źļ Right ‚ÄĒ and that intent actually rules out the "parent count > 1" shortcut I
  floated. Worth flagging because the shortcut almost works and could quietly
  slip into an implementation.

  The shortcut only catches multi-parent at the region node itself. But a region
   node can have a single parent and still be reachable from multiple roots, if
  the fan-out happens above N:

     R1        R2
      \        /
       \      /
        A  (U-owned, parents R1 and R2)
        |
        N  (U-owned)         ‚Üź N has one parent (A), passes precondition 2
        |
        P  (U-owned)
        |
        T  (U-owned, leaf)

  Every node in the region {N, P, T} has exactly one parent. The shortcut says
  "not shared." But each of them is reachable from both R1 and R2 via A, so
  deleting them silently strips that subtree from R2's tree as well ‚ÄĒ the exact
  destructive cross-tree effect the precondition exists to prevent.

  So the check has to be the literal one: for each node in the deletion region,
  gather its set of ancestor-roots; if any node's set has cardinality > 1,
  block. (Halt-frontier other-owned children remain excluded ‚ÄĒ they're detached,
   not destroyed.)

  Equivalently: walk up from each region node and look for any ancestor with > 1
   parent. First fan-out you hit means multiple roots above. That's probably the
   implementation shape, but the spec wording should stay as "reachable from
  more than one root" so the property is what's authoritative, not the
  algorithm.

  One follow-on: this means the deletion check needs an efficient "ancestor
  roots of X" query against the live store. Phase 3's invariant module may or
  may not already expose that ‚ÄĒ worth checking before the proposal commits to a
  shape. Want me to look?

‚Źļ End-to-end the mechanism is:

  Request shape (PRD ¬ß6.2, ¬ß15.3):
  PATCH /taxa/{id}
  X-Username: <current-owner>
  { "owner": "<new-username>" }
  name and owner are both optional in the body; either or both may be present in
   one PATCH.

  Validation/auth chain:

  1. identity-resolution middleware (Phase 2)
     X-Username  ‚Üí  null | unregistered | registered (with originalCasing)

  2. authorization-gate (Phase 2)
     reject null and unregistered callers ‚Üí 403 forbidden

  3. PATCH handler ‚ÄĒ load taxon by id (404 if missing)

  4. owner-check
     caller-username equalsCI taxon.owner ?
     no ‚Üí 403 forbidden (non-owner)

  5. acquire write-lock (Phase 4 new)

  6. body validation
     - owner present?
         null  ‚Üí reject (¬ß6.2: "may never be set to null")
         string ‚Üí username-format check (empty / whitespace) ‚Üí 400
     - name present? ‚Üí ¬ß3.4 format check

  7. registry check on new owner
     userRegistry.isRegistered(newOwner) ?
     no ‚Üí reject  ‚Üź status code is a genuine spec gap, see below

  8. resolve casing
     canonical = userRegistry.originalCasing(newOwner)   // e.g. "alice" ‚Üí
  "Alice"

  9. invariant check
     if name also changed: checkRename(snapshot, id, newName) ‚Üí 409 on violation

  10. apply (atomic under write-lock)
      taxon.owner = canonical    // stored in registered original casing
      taxon.name  = newName      // if present

  11. release lock ‚Üí 200

  Key properties this gives you:

  - Unilateral: no consent flow, no notification. The new owner finds out by
  inspecting the taxa (or being told out of band). PRD ¬ß6.2 is explicit about
  this.
  - Owner stored in registered casing: taxon-domain spec line 19 already
  requires owner storage in original registered casing ‚ÄĒ so step 8
  (originalCasing) is load-bearing. Passing through the raw request string would
   be a spec violation.
  - Owner-check uses equalsCI: because the stored owner is whatever the
  registry's originalCasing returned at last write, and the caller's X-Username
  could differ in case, comparison must be case-insensitive (matching the
  registry's identity rule).
  - Atomic with rename: both validations run, both apply, both rolled back
  together if either fails. The write-lock makes this trivial ‚ÄĒ no transaction
  machinery needed.
  - Self-reassignment: owner: "<caller>" passes every check and applies as a
  (canonicalizing) no-op. Probably fine to allow; the spec doesn't forbid it.
  - Previous owner loses rights immediately: at step 10 the in-memory state
  flips. Any subsequent request from them gets¬†403. The current request
  completes because the auth check already passed at step¬†4.

  One spec gap worth surfacing in the proposal (step 7):

  PRD ¬ß15.6 maps status codes:
  - 400 (validation_error) ‚ÄĒ malformed username
  - 403 (forbidden) ‚ÄĒ caller is null or unregistered
  - 404 (not_found) ‚ÄĒ taxon/tree/proposal/change missing (user is not listed)
  - 409 (conflict) ‚ÄĒ invariants, preconditions, "duplicate user registration"

  "New owner is well-formed but not registered" doesn't fit any cleanly. It's
  not the caller, so it's not 403's "request bearing an unregistered username."
  It's not malformed. The new-owner string is conceptually a reference to a user
   that doesn't exist ‚ÄĒ which would map naturally to 404, but PRD ¬ß15.6's 404
  list doesn't include users. And it's not really a conflict either.

  This needs resolving in the proposal ‚ÄĒ pick one and add a details.tag (e.g.
  unregistered-owner) so the spec is unambiguous. My instinct is 400 
  validation_error with tag unregistered-owner ‚ÄĒ it fits the "request body
  references something that doesn't satisfy a stated constraint" pattern,
  parallel to how malformed names land at 400 ‚ÄĒ but it's worth checking against
  your sense of the model. Want to settle that now or punt it into the proposal?

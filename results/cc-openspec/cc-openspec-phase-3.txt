  The keystone framing is doing a lot of work in the brief — this module's API
  is what Phases 4–7 will be coupled to for the next several thousand lines of
  write code. So getting its shape right is the whole game here.

  Where I see the interesting decisions

  A handful of design questions that aren't obvious from the PRD and that I'd
  want resolved before writing the spec:

  1. Snapshot vs. delta API for the invariant module. The brief says "given a
  candidate state or a proposed change" — those are quite different shapes.
  Three plausible styles:

  (a) snapshot:  check(state) -> Ok | Violations          # check a hypothetical
   world
  (b) delta:     check(state, op) -> Ok | Violations      # current + proposed
  mutation
  (c) both:      a thin (b) layered over (a)
  1. Most write paths (add-edge, rename) are naturally deltas. But cascades and
  grafts produce a sequence of deltas applied to evolving state, which is closer
   to (a) on a recomputed snapshot. Choice here shapes everything downstream.
  2. What counts as "the state"? The store could expose taxa-and-edges directly,
   or hand the invariant module a read-only view. Pure-module discipline pushes
  toward "module accepts a graph value, doesn't reach into the store" — which is
   what makes hand-built test graphs work.
  3. Cross-tree checks on rename. Invariant 3 requires "satisfies in every
  containing tree." That means the invariant module needs reachability to
  identify which roots contain the taxon — which is why reachability is item (2)
   in the brief and not folded into invariants. The dependency direction
  matters: invariants depend on reachability, reachability depends on the store
  shape.
  4. Per-tree uniqueness of occurrence (invariant 2). This is the trickiest of
  the three to test cleanly. A diamond across trees is legal; a diamond within
  one is not. The detection is "is there a taxon reachable by more than one path
   from this root?" — distinct from cycle detection. Easy to muddle them in
  code.
  5. GET /trees/{rootId} for shared subtrees. If c2 is shared between trees T1
  and T2, calling GET /trees/T1 expands c2's subtree under T1. Calling GET 
  /trees/T2 expands it again under T2. The shape is per-tree expansion, not a
  deduplicated graph — and the response surely needs to surface "this is shared"
   somehow (the §15.2 fields don't say, but shared is a defined predicate in
  §3.3).
  6. State-seeding for tests. No write endpoints exist, so tests have to
  construct graphs by calling store functions directly. That's a real surface —
  it isn't just "an internal detail" — because every Phase 3 test depends on it.
   Worth designing it deliberately rather than letting it grow ad-hoc.
  7. ID generation + reset. Tiny but real: monotonic counter, reset zeros it.
  The spec needs to say so so Phase 4+ can rely on it.
  8. No requireRegistered calls anywhere in Phase 3. All four read endpoints are
   open. The authorization-gate spec's forward obligation only fires for writes,
   and there are none. Worth being explicit about that in the proposal so the
  gate stays unblemished.

  Where do you want to start?

  A few natural threads:

  - The invariant module's API shape (decision 1 + 2 above) — probably the
  highest-leverage thing to nail down first.
  - The data model and reachability layer — the substrate everything else stands
   on.
  - Read endpoint shape, especially how GET /trees/{rootId} handles shared taxa.
  - Test-seeding strategy — what does a Phase 3 test look like when it needs to
  assert "this graph violates invariant 2"?

  Or somewhere else entirely — what's pulling at you?

⏺ Working through the design. Here are the calls I'm making.

  Phase 3 design — decisions

  Capability split: three new specs

  ┌────────────────────────────────────────────────────────────┐
  │ taxon-domain      Taxon entity, in-memory store, server-   │
  │                   side ID generation, snapshot type,       │
  │                   reachability primitives, /reset hook.    │
  │                                                            │
  │ invariant-engine  Pure module: cycle / in-tree-duplicate / │
  │                   in-tree-name-clash checks, plus thin     │
  │                   delta wrappers (checkAddEdge,            │
  │                   checkRename). Sole owner of the rules.   │
  │                                                            │
  │ taxon-reads       GET /taxa, /taxa/{id}, /trees,           │
  │                   /trees/{rootId}. Open to all callers.    │
  └────────────────────────────────────────────────────────────┘

  Reachability sits in taxon-domain (not its own capability) — both invariants
  and reads consume it, and nothing else will.

  1. Invariant module API — snapshot-based, with two delta wrappers

  // Pure core
  checkInvariants(g: Graph): { ok: true } | { ok: false; violations: Violation[]
   }
  // Submoves exposed for direct unit testing
  checkNoCycles(g)
  checkInTreeUniqueOccurrence(g)   // invariant 2
  checkInTreeNameUniqueness(g)     // invariant 3 (uses equalsCI)

  // Thin "would this op violate" wrappers — synthesize a candidate Graph, call 
  core
  checkAddEdge(g, parentId, childId)
  checkRename(g, taxonId, newName)

  Why snapshot-first: makes hand-built test graphs trivial; matches the "build
  as a pure module" directive. Delta wrappers exist so Phase 4 doesn't have to
  invent the API — and so the keystone discipline ("every write path calls this
  module") is testable now. Delete and detach can't introduce invariant
  violations, so they get no wrappers. Returns all violations (not
  short-circuit) — Phase 4 may want full diagnostics.

  2. Graph value — the contract between store, invariants, and reads

  type TaxonId = string;            // server-assigned; opaque (`t1`, `t2`, …)
  type Taxon = { id: TaxonId; name: string; owner: string };
  type Graph = {
    taxa: ReadonlyMap<TaxonId, Taxon>;
    childrenOf: ReadonlyMap<TaxonId, ReadonlySet<TaxonId>>;
    parentsOf:  ReadonlyMap<TaxonId, ReadonlySet<TaxonId>>;
  };

  Both child and parent indices in the value. Store maintains both atomically;
  tests build either by hand or via a seed() helper. The invariant module
  accepts Graph and never reaches into the store — that's what makes it pure.

  3. Store and ID generation

  src/taxa/store.ts holds byId, childrenOf, parentsOf, and nextId = 1. IDs are
  t${n} strings, monotonic, reset to 1 on /reset. Exposes snapshot(): Graph,
  lookups, and reset(). Also exposes a seed() for Phase 3 test fixtures,
  documented as "Phase 3 only — Phase 4's mutation primitives supersede this for
   non-test use." That keeps the spec talking about behavior given state, not
  how the state got there.

  4. Reachability primitives (in taxon-domain)

  roots(g): Set<TaxonId>
  containingRoots(g, t): Set<TaxonId>   // by walking parents up to root taxa
  isShared(g, t): boolean                 // |containingRoots| > 1
  reachableSubtree(g, top): Set<TaxonId>

  shared is the predicate §3.3 defines; everything else falls out of standard
  graph traversals.

  5. Read endpoint shapes

  All four endpoints are open (no requireRegistered). Response wrappers match
  Phase 2's { users: [...] } style.

  GET /taxa
    → { taxa: [ { id, name, owner, children: [id], parents: [id], shared } ] }

  GET /taxa/{id}
    → { taxon: { id, name, owner, children, parents, shared } }    | 404

  GET /trees
    → { roots: [ { id, name } ] }

  GET /trees/{rootId}
    → { root: <id>, tree: { id, name, owner, shared, children: [...] } }   | 404

  Adding shared to the per-taxon shape — defined in §3.3 and trivially
  derivable; saves clients the math. Nested expansion in GET /trees/{rootId} is
  unambiguous because invariant 2 forbids in-tree duplicates. Children order is
  implementation-defined but stable within a snapshot (insertion order from
  Set).

  6. Forward obligations and reset

  - State-reset forward obligation: discharged. resetAll() gains a taxaReset()
  call that clears all three maps and zeros nextId. New test asserts this.
  - Authorization-gate forward obligation: vacuous. Phase 3 introduces no
  writes.
  - New invariant-engine forward obligation: every later write path SHALL go
  through the invariant module rather than reimplement any of the three §3.3
  rules. Mirrors the authorization-gate pattern — gives Phase 4+ a spec-drift
  signal if anyone reaches around it.

  7. Testing topology

  - test/invariants/ — unit tests against hand-built Graph values. Each
  invariant exercised at boundary conditions (one-edge cycle, self-loop, diamond
   within a tree, diamond across trees that is legal, name clash including
  case-variant, three-deep paths).
  - test/taxa/store.test.ts — ID monotonicity, snapshot purity (mutating
  returned sets must not corrupt the store), reset zeros the counter.
  - test/taxa/reads.test.ts — HTTP tests against seed()-installed graphs,
  exercising each endpoint's shape and 404 paths.
  - test/reset/taxa-reset.test.ts — seed → reset → empty + IDs restart at t1.

  Initial test plan written from the three new specs in the Rule | Input | 
  Expected | Notes table format Phase 2 established; frozen on commit; final
  plan + changelog at phase close.

  ---
  Things I'm choosing not to escalate but want on the record

  - Delta wrappers shipped before they have HTTP callers. Speculative but cheap;
   Phase 4 may refine signatures. Captured as a design risk.
  - shared in response bodies, beyond the brief's listed fields.
  Forward-additive; the cost of removing it later is zero (omit it from the
  response); the cost of adding it later means a behavior change.
  - seed() lives in src/taxa/store.ts. A test-only surface in production source
  is mildly uncomfortable but pragmatic. Marked clearly.

  If any of these don't sit right, say so. Otherwise I'm ready to materialize
  this as /opsx:propose add-phase-3-domain-invariants and you can review the
  artifacts.

  The setup Phase 5 left for us:

  Every derived change already carries the data promotion needs:

  DerivedChange {
    decisionDeps:   [ChangeId]    ← every `add`/graft ancestor on payload path
    existenceDeps:  [ExistenceDep]← every no-op/rename ancestor's claim +
                                    this change's own implicit edge claim
    reviewer:       concrete | deferred(parentChangeId)
    disposition:    "queued" if decisionDeps empty, else "latent"
  }

  So promotion = "transition latent → queued when the gate opens." The gate is
  two-part: decision deps satisfied AND existence deps still hold (§11.3).

  Promotion trigger: accepting an add/graft change.

     Accept C  ─────▶  Find direct dependents of C
                            │
                            ▼
                       ┌─────────────────────────────────────┐
                       │ For each dependent M:               │
                       │   1. Resolve deferred reviewer      │
                       │      (M's reviewer→C.acceptedBy)    │
                       │   2. Recheck existence deps         │
                       │   3. If all hold: latent→queued     │
                       │      (insert into queueByUser)      │
                       │   4. If not: ??? (open question)    │
                       └─────────────────────────────────────┘

  "Direct dependent" — what it actually means.

  Because §11.3 doesn't formally define it, here's what I think it has to mean
  given the data:

  ▎ M is a direct dependent of C iff C is the deepest add ancestor on M's 
  ▎ payload path — i.e., C is the last entry of M.decisionDeps.

  Cleanly: when C is accepted, every prior entry in M.decisionDeps was already
  accepted (otherwise C couldn't have been queued, transitively). So C being the
   deepest means M is now fully unblocked. Anything deeper than C in M's chain
  (other adds nested under C) stays latent until that deeper add accepts. One
  accept = one level of unfurling.

  Data-structure choice: scan all changes vs. maintain a reverse index
  lastDecisionDep: Map<ChangeId, Set<ChangeId>>. Writes are serialized and
  low-volume per §11.1, so scanning the proposal's changes (we have
  changesByProposal) is fine. But if we want O(1) per accept, the index is cheap
   to maintain.

  Existence-dep re-evaluation.

  Each existence dep is { parent: PayloadParentRef | null, childId: TaxonId }.
  To check it at promotion time:

  Resolve parent → live TaxonId:
    { kind: "top" }                  → proposal.topTaxonId
    { kind: "live", taxonId }        → taxonId (must still exist)
    { kind: "to-create", parentChId} → C's minted taxon id (looked up
                                       because we just accepted C above)
    null  (add-graft global claim)   → no parent; childId must exist anywhere

  Then assert:
    - childId taxon still exists in live state
    - parent→childId edge exists in live state (skipped for global)

  The to-create resolution requires that, when accepting an add-create, we
  record both acceptedBy and mintedTaxonId on the change record. That's the same
   record the next promotion will read.

  The data we'll need to start tracking on accepted changes:

  ┌──────────────────┬────────────────┬─────────────────────────────────────┐
  │      Field       │    Set when    │              Used for               │
  ├──────────────────┼────────────────┼─────────────────────────────────────┤
  │ acceptedBy:      │ accept of any  │ (a) deferred reviewer resolution    │
  │ string           │ op             │ for dependents; (b) audit           │
  ├──────────────────┼────────────────┼─────────────────────────────────────┤
  │ mintedTaxonId:   │ accept of      │ (a) to-create payload-parent        │
  │ TaxonId          │ add-create     │ resolution in later accepts; (b)    │
  │                  │ only           │ existence-dep resolution            │
  └──────────────────┴────────────────┴─────────────────────────────────────┘

  These should sit on the DerivedChange shape (with a ?:), or in a parallel
  acceptances: Map<ChangeId, {…}> on the store. Either works; I'd lean toward
  fields on the change for locality.

  The genuine open question: what happens when existence deps fail at promotion?

  §11.3 says promotion happens "provided their existence dependencies also
  currently hold." If they don't, §11.3 implies "stay latent" by silence — but a
   latent change whose existence deps will never recover (the no-op taxon was
  directly deleted) is a zombie. It can never become queued, and §11.4 case 1
  doesn't cover it (case 1 is about decision dep ancestor rejection).

  Three coherent options:

  Option A: Stay latent forever (literal §11.3 reading)
    + Matches PRD wording exactly
    - Zombie state; no way for reviewer or proposer to see "this is dead"
    - Status tree shows latent → proposer thinks "still waiting"

  Option B: Transition to invalid (extend §11.4 case 1 by analogy)
    + Honest about the change being dead
    + Status tree shows invalid with reason
    - PRD silent on this case — we'd be filling in the spec
    - But the result is observably reasonable

  Option C: Defer to Phase 7
    + Lets us narrow Phase 6 scope strictly to brief
    - Means we ship a known zombie hole

  My lean: B, but it's a spec-fill-in. We should resolve it explicitly in the
  proposal and call out that we're extending §11.4 by analogy (existence-dep

  would have to have become a leaf and then been deleted between submission and

  decide deliberately.

  2. For each dependent: resolve deferred reviewer using the parent's

  The interesting thing about reject: it's almost trivial if we read the spec 
  one way, and slightly nontrivial if we read it the other.

  There's a phrasing tension in the PRD worth surfacing before we move on.

  §12.3:  "Reject one queued change. Its state becomes rejected.
           All of its payload descendants become invalid (§11.4 case 1)."

  §11.4 case 1:  "An ADD/GRAFT ancestor on the change's path was rejected
                  or became invalid. All of that ancestor's payload descendants
                  become invalid... Such changes were latent (never queued),
                  so no reviewer action is involved."

  §12.3 says "all descendants" without qualification. §11.4 case 1 restricts to
  add/graft ancestor rejections specifically.

  These reconcile if you read §12.3 as a forward reference to §11.4 case 1
  (i.e., §12.3 is shorthand and case 1 is the rigorous statement). That reading
  is supported by the dep model:

  - A rename rejected → taxon still exists with its old name. Descendants'
  existence deps were on the taxon's existence, not its name (PRD §10.2: "a
  pending rename does not block a descendant"). So descendants are unaffected.
  - A detach rejected → no descendants exist in the first place (detach is a
  leaf in the payload tree by validator construction).
  - An add-create or add-graft rejected → descendants' decision-dep on this
  change can never be satisfied. They cascade to invalid. ✓

  So: only rejecting an add/graft cascades. Rejecting a rename rejects one
  change and does nothing else; rejecting a detach rejects one change with no
  descendants to worry about.

  That's worth resolving explicitly in the proposal — same "raise it explicitly"
   instruction from CLAUDE.md that applied to the
  existence-dep-failure-at-promotion question.

  If we adopt the dep-model interpretation, the implementation is almost free.

  Descendants of an add/graft change C in the payload tree are exactly the
  changes in the same proposal whose decisionDeps contains C — because
  decisionDeps captures every add/graft ancestor on the payload path. This is
  already in the Phase 5 data; no new walk, no new index.

  On reject of change C:
    ─────────────────────
    1. C: queued → rejected
       (set reason: "rejected by reviewer" or include reviewer's reason if API
  takes one)
    2. queueByUser: remove C (it leaves the actionable queue)
    3. If C.op ∈ {add-create, add-graft}:
         For each M in changesByProposal[C.proposalId]:
           if C.changeId ∈ M.decisionDeps:
             M: latent → invalid
             set reason: "decision dependency <C.changeId> rejected"
       (rename/detach reject: no descendants to touch)

  Three small things worth pinning down explicitly:

  1. Does reject take a reason from the reviewer?
  PRD doesn't say. §15.5 lists POST /changes/{id}/reject with no body specified.
   §11.5 says "for negative states (rejected/invalid), an optional
  human-readable reason." So a body of { reason?: string } would be reasonable,
  defaulting to e.g. "rejected by reviewer" when omitted. Or we punt and always
  use a fixed reason for Phase 6. Either's fine — small UX call.

  2. What about state transitions of descendants that weren't latent?
  With the dep-model interpretation, descendants are all latent (they were
  waiting on C). But guardrail: if for some reason a descendant is in a
  different state (e.g., already invalid via another path), we should be
  defensive. Simplest: only transition latent → invalid. If it's already

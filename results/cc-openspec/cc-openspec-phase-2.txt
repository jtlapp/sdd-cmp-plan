  Open threads I'd want to chew on before proposing

  1. Identity resolution vs. authorization gate — one middleware or two?

  The brief talks about "a reusable gate, applied by later phases to write
  endpoints." But GET /queue (Phase 7-ish) is a read that still needs caller
  identity. Suggests two distinct concerns:

  - Identity resolution — runs everywhere, produces { kind: "null" } | { kind: 
  "registered", username }, throws on malformed/unregistered.
  - Authorization gate — selectively attached to writes; just asserts "kind ===
  registered".

  Or you fold them into one and accept that "kind" gets read by handlers
  directly. Worth picking a stance.

  2. What does "open" mean for POST /users and GET /users?

  The PRD gives us three exemption levels and doesn't fully disambiguate:

  ┌─────────────┬──────────────────┬────────────────┬──────────────────────┐
  │  Endpoint   │  Header format   │  Registered    │        Source        │
  │             │      check?      │     check?     │                      │
  ├─────────────┼──────────────────┼────────────────┼──────────────────────┤
  │ POST /reset │ No (explicit)    │ No (explicit)  │ state-reset spec is  │
  │             │                  │                │ clear                │
  ├─────────────┼──────────────────┼────────────────┼──────────────────────┤
  │ POST /users │ ?                │ No (open,      │ implied              │
  │             │                  │ §15.1)         │                      │
  ├─────────────┼──────────────────┼────────────────┼──────────────────────┤
  │ GET /users  │ ?                │ No (open,      │ implied              │
  │             │                  │ §15.1)         │                      │
  ├─────────────┼──────────────────┼────────────────┼──────────────────────┤
  │ writes      │ Yes              │ Yes            │ §4                   │
  │ (later)     │                  │                │                      │
  └─────────────┴──────────────────┴────────────────┴──────────────────────┘

  My read: header format check is global by default, with /reset as the explicit
   exemption. So POST /users with X-Username:    → 400. But this is genuinely
  under-specified — worth resolving in the proposal, not at code-write time.

  3. How do we "prove the gate in isolation"?

  Phase 2 has no protected endpoints yet — they're explicitly deferred to phases
   3–7. Three options:

  - Mount the middleware on a stub/fixture route in tests only.
  - Add a temporary POST /_test/gated endpoint behind a flag (and rip out in
  Phase 3). Feels ugly.
  - Test the middleware as a pure function/handler without a route at all.

  Option 1 feels cleanest. But it raises: do we want the gate to be a Hono
  middleware, or a small helper that endpoints invoke explicitly? Affects
  testability.

  4. Registry storage — case-insensitive index, case-preserving display.

  Phase 1's username spec already commits to "no normalized form SHALL be
  stored." So the registry needs to store original casing but de-dupe
  case-insensitively. Probably Map<ciKey, originalCase> or a Set<originalCase>
  plus a Set<ciKey> for membership. Trivial, but worth deciding once so the spec
   can state it.

  5. Body-username on POST /users reuses the same validator as the header.

  So one validator, two consumption sites. The spec should note this explicitly
  so we don't drift into "header rules" vs "body rules."

  6. /reset extension.

  Forward obligation from Phase 1's state-reset spec: Phase 2 extends /reset to
  clear the registry, and ships at least one test for it. Already locked in by
  the existing spec — just need to remember it.

  ---
  Which of these do you want to dig into first? Or is there something the brief
  glosses that I haven't surfaced?

  Key shift: the middleware doesn't enforce "must be registered" — it just 
  resolves identity and 400s on malformed. The enforcement lives in a tiny
  requireRegistered(ctx) helper that future write endpoints call. That's still
  "one middleware" — the middleware does the universal work; the helper does the
   per-endpoint work.

  On your question #3 — "do we still have a problem? do we still have a gate?"

  Yes to both, but smaller than I painted them:

  The gate is now two things working together:
  - The middleware (rejects malformed headers; classifies caller).
  - The requireRegistered helper (rejects null + unregistered for write
  endpoints).

  The "isolation" problem mostly evaporates. We already have three real
  endpoints in Phase 2 (POST /users, GET /users, POST /reset) that exercise the
  middleware end-to-end — that gives us the integration coverage for the parser
  half of the gate. For the helper half, we just unit-test the function
  directly. No fixture endpoint needed, no test-only scaffolding shipped.

  So the phase-2 deliverable for "the gate" is really:
  1. Middleware that runs globally (except /reset), classifies, 400s on
  malformed.
  2. A requireRegistered(ctx) function that throws 403 if caller is null or
  unregistered.
  3. Tests that hit (1) through real endpoints and (2) as a unit.

  Phase 3 is when (2) actually gets wired into HTTP handlers.

  One small thing your "whatever" leaves open

  Should POST /users and GET /users still validate the header format (and 400 on

   is fully exempt. That keeps the contract clean: anywhere you send X-Username,
   you must send it well-formed. Want me to lock that in, or do you want those
  two endpoints fully exempt like /reset?

⏺ Pausing — implementation revealed an HTTP transport reality the spec doesn't
  acknowledge.

  What I found

  Hono's test transport (and any RFC-7230-compliant HTTP server) normalizes 
  header values: leading/trailing ASCII whitespace (space, HTAB, newline) is
  stripped before the handler sees it. From the diagnostic:

  sent="   " (whitespace-only)  got=""        ← becomes absent
  sent=" alice" (leading SP)    got="alice"   ← stripped
  sent="alice " (trailing SP)   got="alice"   ← stripped
  sent="\talice" (leading TAB)  got="alice"   ← stripped
  sent="alice\n" (trailing LF)  got="alice"   ← stripped
  sent="\u00a0alice" (NBSP)     got="\u00a0alice"  ← survives (non-OWS)

  Implication: the spec's "malformed X-Username" scenarios using ASCII
  whitespace can't be exercised through real HTTP — those bytes never reach my
  middleware. The middleware does reject them correctly when it sees them; HTTP
  just rarely delivers them.

  The spec's behavioral contract still holds:
  - absent / empty → null caller ✓
  - well-formed registered → registered caller ✓
  - well-formed unregistered → unregistered caller ✓
  - present, non-empty, malformed → 400 validation_error (still true; just hard
  to reach via HTTP)

  What I'd do

  Three small adjustments, none of them spec-defeating:

  1. Update the test scenarios to use NBSP (\u00A0) or other non-OWS whitespace
  as the HTTP-level "malformed" probe — those survive transport and trigger the
  middleware's 400 path through real HTTP. (My existing HTTP failures all use
  ASCII whitespace.)
  2. Add a direct unit test of the middleware function with a mock context,
  asserting that when called with a string like "   " or " alice" it throws
  ApiError("validation_error"). This proves the contract holds for any input,
  not just HTTP-reachable ones.
  3. Add one paragraph to design.md (or as a new Decision 10) acknowledging that
   HTTP OWS-stripping makes ASCII-whitespace malformed cases effectively
  unreachable, and that the middleware's correctness for those inputs is
  verified at the unit-function level.

  The spec text itself doesn't need to change — its requirements describe what
  the middleware does, not what HTTP delivers.

  Want me to proceed with that adjustment? Or would you rather I rework the spec
   to drop the ASCII-whitespace scenarios?

  The "frozen world" property of this phase is important: after POST /proposals
  returns, nothing about a proposal changes. There is no path in this phase that
   transitions a change between states. Phase 6 introduces all the dynamics.

  Three things that pulled my attention reading ¬ß8‚Äď¬ß11 against the brief

  1. Existence checks have a curious status at submission.

  ¬ß9.2 explicitly requires the top taxon to be currently in the target tree.
  ¬ß10.2 then frames every ancestor no-op/rename as an "existence dependency" ‚ÄĒ
  language that suggests evaluated against live state at some moment, not
  validated up front. And ¬ß11.3 only mentions decision dependencies as the
  queueing gate. ¬ß11.4 says "validity is evaluated lazily."

  Read together I think the design is:
  - Top taxon: hard-rejected at submission (the only existence guarantee ¬ß9.2
  demands).
  - Other operative nodes: their existence is recorded as a dependency,
  evaluated only when the change is acted on later (Phase 6+).
  - So Phase 5 can submit a proposal renaming a taxon that doesn't currently
  exist; it just sits there with a dependency that will never resolve.

  But that also forces a routing question: rename routes to "owner of the 
  renamed taxon." If the taxon doesn't exist at submission, there is no owner to
   record. Does the change route to "no one" until/unless the taxon appears? Or
  do we resolve routing at submission against a snapshot? ¬ß11.4 case 3 suggests
  routing isn't re-evaluated after submission ‚ÄĒ an owner change is an
  invalidation, not a reroute. Which would imply routing is frozen at submission
   and a missing taxon at submission ‚áí routing is undefined.

  The PRD doesn't say explicitly. I'd want to resolve this before specifying ‚ÄĒ
  possibilities:
  - (a) Routing snapshot at submission. Missing-target ‚Üí reject the proposal (or
   just that change?) at submission.
  - (b) Routing also evaluated lazily, like existence. The change has no
  reviewer until the taxon exists.
  - (c) Routing snapshot, but missing-target produces an immediately-invalid
  change rather than a submission-time error.

  2. What's a "shape" error vs. a "dependency" error?

  The brief leans on the four-op table from ¬ß9.1. Some of its rules clearly bind
   at submission (e.g., detach must be a leaf; add needs either id-null+name or
  id+no-name). Others are runtime-flavored (does the payload-parent currently
  have the named child? does an existing-id add refer to a real taxon?). The
  line between "malformed payload ‚Üí 400" and "well-formed but its existence
  dependency will never resolve ‚Üí submitted as latent/invalid" is not drawn
  explicitly in the PRD.

  A working interpretation:
  - Shape (400 at submission): field-rule violations from ¬ß9.1, top-taxon not
  no-op/rename or not in target tree, detach-not-a-leaf, payload tree itself
  malformed, missing required fields, name-shape violations (¬ß3.4).
  - Existence (recorded as dep, not blocking): target of a rename/no-op/detach
  not present at expected position; graft id refers to a non-existent taxon.

  But that second bucket ‚ÄĒ especially "graft id refers to nothing at all" ‚ÄĒ
  feels like it should be 400, not a silently-recorded dep that can never
  fulfill. Worth deciding deliberately.

  3. The status view's "isomorphic to payload" plus create-IDs.

  ¬ß11.5's status view is the payload tree annotated with dispositions. For
  creates (id: null), the payload has no taxon ID ‚ÄĒ but the proposer still needs
   to refer to the corresponding change (per GET /queue's "change ID"
  requirement, and per POST /proposals returning a status tree). So at
  submission we'd assign change IDs to every operative payload node, return them
   in the status tree, and surface them in /queue. No-ops have no change ID
  (they're structural).

  We probably also need a stable way to refer to a payload position even for the
   operations that DO have a taxon ID ‚ÄĒ because a rename of c3 and a detach of
  c3 in the same payload would be two different changes against the same id.

  Threads worth pulling

  1. The "validation vs forbidden" hairline crack (§4)

  This phrase from the brief — "surfacing the validation-vs-forbidden 
  distinction" — is doing more work than it looks. The PRD draws three distinct
  cases:

     incoming X-Username:
     ┌──────────────────┬──────────────────┬─────────────────────┐
     │ absent / empty   │ present, well-   │ present, malformed  │
     │                  │ formed, unreg.   │ (ws-only, padded)   │
     ├──────────────────┼──────────────────┼─────────────────────┤
     │  null user       │  forbidden (403) │  validation_error   │
     │  reads allowed   │  unregistered    │  (400)              │
     │  writes blocked  │                  │  NOT silently null  │
     └──────────────────┴──────────────────┴─────────────────────┘
                                ↑
                    Phase 2 boundary (registry)

  Phase 1 ships only the format validator — it can't know "unregistered" yet.
  The interesting question is: what shape does the validator have so Phase 2 can
   layer the registry check on cleanly?

  A few sketchable options:

  A)  validateUsername(s: string): { ok: true } | { ok: false, reason: ... }
      // pure format check; middleware composes registry check separately

  B)  parseUsername(header: string|undefined):
         { kind: "null" } | { kind: "malformed", reason } | { kind: "named",
  name }
      // validator owns the "absent vs malformed vs candidate" split

  C)  classes / discriminated union with explicit "candidate" state
      that Phase 2's middleware promotes to "registered"

  (B) is more opinionated and pulls the trichotomy into Phase 1 even though only
   two of the three cases produce validator-visible output. (A) is minimal but
  pushes the trichotomy into the Phase 2 middleware. There's a real choice here
  about where the spec lives.

  2. Case-insensitive comparison — predicate or normalizer?

  §3.4 and §4 both say "compared case-insensitively." Two paths:

     a)  normalize(s) → string    (e.g. toLowerCase)
         used wherever uniqueness/equality is checked

     b)  equalsCI(a, b) → boolean
         no normalized form leaks out

  (a) makes Phase 3's per-tree uniqueness check trivially a Set lookup but
  commits to a canonical form being storable. (b) keeps the original casing the
  only form anywhere in the system, but makes uniqueness checks an O(n) scan
  unless you build a side index.

  Worth pinning before Phase 3, but the validator's surface (does it expose a
  normalizer? equality? both?) gets decided now.

  3. The error helper — class, function, or middleware?

  §15.6 is rigid: { error: { code, message, details? } }, with a fixed
  code→status mapping. Every later phase must emit through this. Three flavors:

     throw new ApiError("conflict", "in-tree name clash", {...})
     → caught by Hono onError → emitted as envelope

     return respondError(c, "validation_error", "...")
     → explicit per-callsite, no throwing

     c.json(err("conflict", "...", {...}), 409)
     → builder + manual status

  The choice ripples through every endpoint in P3–P6. Throwing is ergonomic in
  deep code paths (validators returning into handlers), but Hono's middleware
  story changes how you'd wire it. Worth picking before we have callsites.

  4. The reset endpoint at Phase 1 is almost a no-op — but it's a contract

  ▎ "implement it as a true full reset so that, as each later phase adds state, 
  ▎ that state falls under reset's fresh-start guarantee"

  That's an extensibility statement, not a behavioral one. The shape is: a
  single resetAll() that each phase contributes to. Either a registry of
  reset-handlers, or a single function that imports each subsystem's reset. Tiny
   architectural decision, but compounding — by Phase 7 you have ~5 subsystems
  to clear.

  5. Initial test plan — granularity?

  The brief is explicit that the initial plan is frozen the moment
  implementation starts, with a separate final plan that diffs from it. The
  value of the artifact is the anticipated-vs-discovered delta. That only works
  if the initial plan is specific enough to be wrong about.

     Too coarse: "validate taxon names per §3.4"        ← can't be wrong
     Too fine:   one bullet per assertion line          ← brittle
     Sweet spot: per-rule × per-disposition matrix
                 (e.g. "empty name → validation_error",
                       "leading whitespace → validation_error",
                       "case-variant equality → equal",
                       "trailing tab → rejected" …)

  Worth deciding what "specific enough" looks like before writing it.

  6. Subtle PRD bit: §3.4 says the validator rule and §15.6 says the error code,
   but nothing says the validator returns the §15.6 code

  The natural mapping is obvious — but is the validator in the business of
  producing §15.6 codes, or just producing yes/no/why, and the error helper maps
   that to a code? Coupling decision that determines whether the validators are
  reusable in a non-HTTP context.

  Smaller things now in the open

  These mostly fall out of the choices you just made:

  a) Shape of reason

  reason is the only thing the validator says about failure. It has three
  readers:
  - humans (eventually shown in message)
  - tests (assertions key off it — and your "sweet spot" granularity needs
  distinguishable values)
  - future code that might want to react to specific failure modes (probably
  never, but…)

  Three flavors:

   plain string          "name must not begin or end with whitespace"
   tag + string          { tag: "leading-whitespace", message: "..." }
   tag only              "leading-whitespace"   (test-friendly, message
  reconstructed)

  The middle one is most flexible. The tag-only is most test-friendly but pushes
   message construction into the throw site. Tag + string lets you assert on tag
   in tests while letting the throw site decide whether to surface the message
  verbatim or wrap it.

  b) Reset extensibility pattern

  Three shapes, increasing in commitment:

   1. Each subsystem exports `reset()`. P1's /reset handler imports
      each one and calls them in order. Phase N adds an import line.

   2. resetRegistry: a module where subsystems call register(fn) at
      module-load. /reset iterates the registry. No imports needed.

   3. Single state-owning module that exposes mutate operations and
      a single `clearAll()`. All in-memory state lives in one place.

  (1) is dumbest and works. (2) is decoupled but spooky (registration is a side
  effect of import). (3) fights the natural per-phase encapsulation. I'd default
   to (1) unless there's a reason not to — but it's your call.

  c) equalsCI placement

  Both validators use it (well, the comparison rule applies to both). Three
  obvious homes:

   src/format/compare.ts          ← shared utility
   src/format/taxon-name.ts       (and re-exported from username.ts)
   inside each validator module   (duplicated)

  The first is the only one that doesn't fight itself.

  d) Non-ApiError throws → 500

  onError should also catch anything that isn't an ApiError and emit the §15.6
  envelope with code "internal_error" (or similar — not listed in §15.6, so a
  new code) at 500. Otherwise the envelope contract leaks at unexpected throws.
  Worth mentioning in the spec since it's the system invariant, not just the
  happy path.

  e) Health endpoint name

  The brief just says "a health/version endpoint." GET /health? GET / returning
  { ok: true, version: "0.1.0" }? Trivial but the spec should name it.

  f) Test-plan markdown shape

  The "sweet spot" granularity is going to be a table-shaped artifact. Worth
  deciding the column convention now since it'll be reused across 6 phases:

   | Rule           | Input             | Expected            |
   |----------------|-------------------|---------------------|
   | non-empty      | ""                | { ok: false, ... }  |
   | no lead ws     | " foo"            | { ok: false, ... }  |
   | case-insens.   | equalsCI("A","a") | true                |
   | …

  Or one section per rule with prose. Up to you, but the format Phase 1 chooses
  is the template every other phase inherits.

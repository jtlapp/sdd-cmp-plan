  Plan

  Scope this phase ships (and only this):

  1. In-memory taxon store — src/taxa.ts. Taxon = { id, name, owner, childIds: 
  Set<id> }; parent set derived. Server-side ID generator (monotonic,
  implementation-defined per §2). clear() wired into the existing registerReset
  seam so POST /reset drains taxa + the ID counter. Exports fixture-only 
  construction primitives (createTaxonFixture, attachChildFixture) that
  deliberately do not enforce invariants — Phase 4 will build validating
  wrappers. No HTTP write endpoint here.
  2. Reachability layer — src/reachability.ts. Pure functions: roots(state),
  descendants(taxonId, state), treesContaining(taxonId, state) (upward walk to
  roots), isShared(taxonId, state) (treesContaining.length > 1), plus an
  in-tree-traversal helper that visits each reachable taxon and surfaces
  duplicate-path detection. No HTTP/storage coupling.
  3. Invariant module — src/invariants.ts. Pure, no I/O. API:
    - checkNoCycles(state)
    - checkPerTreeUniqueOccurrence(state) (§3.3 #2 — diamonds within one tree)
    - checkPerTreeNameUniqueness(state) (§3.3 #3, case-insensitive per §3.4)
    - evaluateAll(state) → structured pass/fail naming the offending
  taxon/edge/root/clashing name, sufficient for Phase 4 to surface as 409 
  conflict reasons.
  Operates on a whole state (Phase 4 produces the candidate state and
  re-evaluates — see question G below).
  4. Read endpoints — src/routes/reads.ts, mounted in app.ts. All four use
  identityWithRegistry; null can read (§15.2). 404 via the existing envelope for
   missing ids:
    - GET /taxa → { taxa: [...] }
    - GET /taxa/{id} → { id, name, owner, childIds, parentIds }
    - GET /trees → { trees: [...] }
    - GET /trees/{rootId} → nested { id, name, owner, children: [ ... ] }
  5. Two test-plan artifacts (test-plans/phase-3-initial.md frozen before
  coding, phase-3-final.md with changelog at the end), and tests at four levels:
   store/ID/reset unit tests, reachability unit tests on hand-built multi-tree
  DAGs, invariant module unit tests including each violation, and read-endpoint
  tests through createApp. Run command stays npm test (tsc && node --test 
  "dist/**/*.test.js").

  Out of scope (deferred per the brief): no mutation endpoints, no
  proposals/queues, no §15.3–§15.5 surface.

  Spec items I'd have to decide — flagging before I code

  PRD §15.2 names the fields but doesn't pin the JSON shapes. My defaults below
  match Phase 2's style; tell me if any of these should differ.

  - A. GET /trees/{rootId} shape. Default: recursive nested — { id, name, owner,
   children: [ {…}, … ] }. Alternative: flat list of all taxa in the tree.
  (Recursive is well-defined inside a single tree because §3.3 #2 forbids
  in-tree duplicates; for tests that deliberately violate that, I'd document the
   chosen traversal behavior — see G.)
  - B. GET /trees per-entry shape. Default: same field set as GET /taxa/{id} (so
   a root is just a taxon, with parentIds: []). Alternative: a smaller { rootId,
   name, owner }.
  - C. /trees/{id} where the id names an existing non-root taxon. Default: 404 
  not_found (the /trees/ namespace is rooted at roots, not arbitrary subtrees).
  Alternative: 200 returning its subtree.
  - D. Ordering. Default: insertion order (taxa creation order; child IDs in
  attachment order), matching Phase 2's registry ordering choice. Per-taxon
  child order is "unordered" in the model (§3.1) but JSON must pick one —
  flagging.
  - E. Taxon ID format. Default: opaque monotonic strings, e.g. "t1", "t2".
  Implementation-defined per §2; tests don't assume specific IDs (per the
  toolchain supplement).
  - F. Fixture primitives bypass invariants. Required to let invariant tests
  construct deliberately invalid graphs (cycle, in-tree diamond, in-tree name
  clash). Phase 4 will wrap these with validation. Flagging so the construction
  API isn't mistaken for the future write API.
  - G. Invariant-module API surface. The brief says "given a candidate state or
  a proposed change." Default: state-based only (Phase 4 materializes state + 
  change and calls evaluateAll). Alternative: also expose change-shaped probes
  (wouldAddEdgeViolate, etc.) now. State-only is simpler and keeps the module
  purely structural; happy to add change-shaped probes if you want them in
  Phase 3.
  - H. shared predicate exposure. Default: not on the read surface — §15.2's
  field list doesn't include it. Used internally and exported from the
  reachability module for Phase 4 (deletion preconditions). Flagging because
  §3.3 names shared as part of the domain, but §15.2 doesn't surface it.

  Tell me which of A–H to flip, and I'll write the frozen phase-3-initial.md
  first, then implement.

⏺ The substance of issue 4 is how the cascade's frontier expands dynamically as 
  it runs — specifically, that ownership transfers caused by mid-cascade accepts
   can pull newly-routed changes into the same cascade. I want to confirm this
  before coding because it determines whether the implementation walks the
  subtree iteratively (applying → promoting → reconsidering) or precomputes the
  set up front.

  The mechanic

  Three rules collide:

  - §10.1: an add (create/graft) is routed to the owner of the payload-parent.
  - §12.1: accepting an add-create transfers ownership of the new taxon to the
  accepting reviewer.
  - §12.2: the cascade accepts "any newly-queued dependent also routed to this 
  same reviewer."

  Together: when the cascade accepts an add-create, the new taxon is now owned
  by the cascading reviewer. Any nested add/detach whose payload-parent is that
  new taxon therefore routes to the same reviewer once promoted — and joins the
  cascade.

  Worked example (this is Appendix A.1's Bob branch)

  Cascade rooted at Urban Fantasy create, queued to Bob:

  1. Accept Urban Fantasy. Bob becomes its owner. Promotion fires for direct
  dependents.
  2. Paranormal Romance is an add-create whose payload-parent is the
  just-created Urban Fantasy (Bob-owned). It's routed to Bob → joins the
  cascade.
  3. Accept Paranormal Romance. Bob becomes its owner too. Recurse — no further
  dependents in this case, so cascade ends.

  If Paranormal Romance had been an add whose payload-parent was a different
  existing taxon owned by Erin, it would have routed to Erin, become a boundary,
   and stayed queued for her.

  Implementation consequence

  This pushes me toward the iterative implementation (accept seed → run
  promotion → look at what just got queued to this reviewer → continue) rather
  than precomputing the set. Precomputation would have to simulate the ownership
   transfers anyway, so iteration is simpler and matches §12.2's bullet wording
  verbatim. Atomicity comes from the rollback log, not from precomputation.

  What I want confirmed

  - Yes, the cascade frontier expands through ownership transfer (Appendix A.1
  behavior is normative intent).
  - Iterative apply-then-promote-then-check is the right implementation shape.
  - Cascade boundaries are still set by routing-to-other-owners at the moment
  promotion happens (not at cascade start).

  Anything in there you'd push back on?

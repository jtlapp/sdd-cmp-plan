  Plan

  Module layout

  - src/proposals.ts — in-memory store of proposals + per-reviewer queue index;
  pure routing/dependency/disposition logic; registers itself with state.ts for
  /reset.
  - src/routes/proposals.ts — POST /proposals (§15.4), GET /proposals (§15.2,
  optional ?proposer=), GET /proposals/{id} (§11.5).
  - src/routes/queue.ts — GET /queue (§15.2), the calling user's queued changes.
  - Wire all three into src/app.ts.

  Data model (in memory)

  - Proposal { id, proposer, targetRootId, topTaxonId, payload, changes }.
  - payload is the parsed-and-normalized payload tree (we assign each node an
  internal node-id so the status view can be rendered isomorphically).
  - Each operative node (rename/add/detach) gets a Change { id, proposalId, op, 
  payloadPath (top→here, by internal-id), 
  affectedTaxonId/proposedName/payloadParentTaxonId, state, reviewer? }.
  - Disposition per §11.3: a change is queued if no add/graft ancestor sits on
  its payload path; otherwise latent. Nothing is promoted in this phase.
  - Reviewer per §10.1: rename → owner of id; add/detach → owner of
  payload-parent's taxon. For queued changes reviewers are always knowable now;
  for latent changes whose payload-parent is an add-create, the reviewer is
  deferred (the accepting reviewer becomes the owner — Phase 6+). We don't need
  to record reviewers for latent changes in this phase since /queue only
  surfaces queued ones.
  - no-op taxa produce no change; status view marks them structural.

  Submission-time validation

  In this order, mapped to the §15.6 error codes already used by Phase 4:
  1. Identity gate (identityWithRegistry + requireWriter) — same as Phase 4
  writes.
  2. Body parsing — { targetRootId, topTaxonId, payload } shape → 400.
  3. targetRootId must be an existing root → 404 / 400 as appropriate.
  4. payload.id must equal topTaxonId; top op must be no-op or rename (§9.2); id
   must reference a taxon currently reachable from targetRootId (§9.2) →
  400/404/409.
  5. Per-node schema (§9.1): required op/id/name per the table; detach must be a
   leaf; add-create requires name; add-graft ignores name.
  6. Every non-create id must reference an existing taxon (otherwise we can't
  route). I'm treating this as the minimum needed for §10.1 to be well-defined;
  deeper existence-dependency checking is acceptance-time (§10.3, deferred).
  7. Inside the write lock (§11.1): assign proposal-id + change-ids, compute
  payload paths and decision-vs-existence dependency classification, compute
  initial disposition per §11.3, install queued changes into per-reviewer
  queues.

  Tests (proposed file split, mirroring Phase-4 style)

  - proposalsParse.test.ts — schema, top-taxon constraints, partial-payload
  rule, malformed bodies.
  - proposalsRouting.test.ts — per-op routing including the boundary case where
  a rename and its add/detach siblings route to different owners.
  - proposalsDependencies.test.ts — decision-dep vs existence-dep classification
   along the payload path (the A.1 keystone: a rename ancestor does not block a
  descendant).
  - proposalsDisposition.test.ts — initial latent/queued, including nested
  creates (Appendix A.1 Paranormal Romance shape) and grafts with nested ops
  (A.2 shape).
  - proposalsStatusView.test.ts — GET /proposals/{id} is isomorphic to the
  submitted payload, dispositions correct, no-op shown as structural.
  - queueRead.test.ts — GET /queue returns only queued changes for the caller;
  auth and isolation across users.
  - proposalsAuth.test.ts — null / unregistered / malformed-X-Username against
  POST /proposals and reads.
  - proposalsReset.test.ts — /reset clears proposals, queues, and change-id
  counter.
  - proposalsIntegration.test.ts — multi-step end-to-end stories (mixed-owner
  routing, graft preserves ownership in dependency calc, partial payload not
  asserting unlisted siblings).
  - Plus envelope-shape assertions woven through, as in Phase 4.

  Two artifacts to ship: test-plans/phase-5-initial.md (frozen at first commit
  of impl) and test-plans/phase-5-final.md (delivered at the end, with a
  changelog vs. the initial).

  Spec items I'd like your call on before I start

  1. topTaxonId field name. §15.4 and the brief both say topTaxonId. Appendix
  A's examples use topNodeId. I'll use topTaxonId as the on-the-wire field and
  treat the appendix as a non-normative typo. OK?
  2. Must payload.id === topTaxonId? Not stated. I'll require equality and 400
  otherwise — they name the same taxon and a mismatch is ambiguous.
  3. Submission-time "id must exist" check. I'll require every non-add-create id
   (rename, detach, no-op, graft) in the payload to reference a
  currently-existing taxon (404 if not), because routing per §10.1 needs an
  owner. Deeper existence-dependency satisfaction (the no-op/rename ancestor's
  id being at the right position in the target tree) I'll defer to
  acceptance-time per §10.3, since the brief explicitly defers all
  decision/promotion/invalidation behavior. Acceptable scope?
  4. Disposition value for no-op in the §11.5 status view. §11.5 calls for "a
  disposition indicating they are structural and produce no change" without
  naming it. I'll use the literal string "structural" to keep it disjoint from
  the five real states.
  5. POST /proposals 201 response body. §15.4 says "the proposal ID and its
  initial status tree." I'll return { id, proposer, targetRootId, topTaxonId, 
  payload: <status tree> } — the same shape GET /proposals/{id} returns.
  6. GET /proposals filter. §15.2 mentions "optionally filterable by proposer."
  I'll support ?proposer=<username> (case-insensitive, matched via the
  registry's canonical form).
  7. GET /queue in Phase 5. §15.2 says "including those in the
  invalid–awaiting-dismiss sub-state" — that sub-state can't yet arise (no
  acceptances in this phase), so Phase 5's queue is just truly queued changes.
  Phase 7 will add the invalid-awaiting-dismiss case.

  If any of those go against what you'd prefer, say the word and I'll adjust
  before writing the initial test plan.

  Phase 6 plan

  Two new endpoints (new src/routes/reviews.ts)

  - POST /changes/{id}/accept — §12.1, §15.5
  - POST /changes/{id}/reject — §12.3, §15.5

  Both run under the existing writerGates (identity → requireWriter) and
  withWriteLock. Resource resolution order: 404 if the change id is unknown →
  403 if caller ≠ change.reviewer → 409 if change is not currently queued.

  Core review logic (new src/review.ts)

  Acceptance applies the change tentatively, runs evaluateAll(getState()) on the
   candidate state to honor §10.3 (the existing module already iterates every
  tree, which covers the cross-tree clauses for renames and grafts), and rolls
  back on any violation. On success it transitions state = "accepted", dequeues
  it from the reviewer's queue, and runs promotion for direct payload-children.
  Per op:

  - rename → mutate taxon.name.
  - add-create → mint a new taxon with owner = accepting reviewer (§5/§12.1
  ownership transfer), attach under payload-parent.
  - add-graft → attach existing taxon under payload-parent (no ownership change,
   §9.3).
  - detach → remove the parent→child edge.

  Reject simply transitions state = "rejected", dequeues it, and (only when the
  rejected change is an add/graft) recursively marks every payload descendant
  invalid and removes them from any queues they sit in — see ambiguity #1 below.

  Promotion (§11.3) at acceptance time

  On accepting an add/graft, walk its direct payload-children. For each
  operative child whose nearest decision-dep ancestor was this accepted change,
  transition latent → queued and resolve its reviewer:
  - rename → owner of renamed taxon (already known).
  - add/detach whose payload-parent is the just-created taxon → the accepting
  reviewer (since they now own it). Otherwise → owner of the payload-parent
  taxon.

  Push the promoted change onto its reviewer's queue in change-id order (the
  existing single ordered list per reviewer already gives "submission order"
  naturally because change ids were assigned at submission).

  Small support changes in src/proposals.ts

  - Expose Map<ChangeId, PayloadNode> (or store a payloadNode ref on Change) so

  - A dequeue(changeId, reviewer) helper, since
  acceptance/rejection/invalidation all need to remove from the per-reviewer

⏺ Good catch on #3 — "leave latent" is wrong. The reasoning: once an
  existence-dep position is broken (the named taxon was deleted, or detached out
   of the target tree), there's no path back. The proposer's plan is anchored at
   a specific in-tree position; even if someone later re-grafts the taxon, the
  system can't know that's "the same plan." So a latent descendant whose
  existence dep has broken can never legitimately become queued.

  I'll invalidate it instead: state = "invalid" (no queue impact since it was
  never queued). This is symmetric with §11.4 case 1's "they can never satisfy
  their dependencies" reasoning. It isn't case 2 or case 3 (both of which are
  about queued changes — the explicitly-deferred bits), so it doesn't trespass
  on Phase 7.

  So the updated rule at promotion time, walking the just-accepted add/graft's
  direct payload-children:
  - existence deps hold → promote to queued, resolve reviewer
  - existence deps don't hold → mark invalid, recursively invalidate that
  subtree's payload-descendants (same shape as the rejection cascade in #1)

  Plan

  1. Registry (§15.1) — src/registry.ts

  - In-memory store: Map<lowercase, canonical-form>. Lookups case-insensitive
  (§4); store original casing for the listing.
  - register(name), isRegistered(name), list(), clear().
  - Wires into registerReset(clear) so POST /reset empties it (confirms the
  phase brief's "reset clears the state this phase adds").

  2. Routes (§15.1) — src/routes/users.ts

  - POST /users — body { username }. Validates body with the existing
  validateTaxonName-style approach but using parseUsername's valid rule (§4
  format). Malformed body → 400 validation_error. Duplicate (case-insensitive) →
   409 conflict. Success → 201 with the canonical form echoed back.
  - GET /users — returns the registered list. Open to all including null.

  3. Identity middleware (§4) — src/middleware/identity.ts

  Per-request: parse X-Username, then resolve to one of:
  - {kind: "null"} (absent/empty)
  - 400 validation_error (malformed format)
  - 403 forbidden (well-formed but not in registry)
  - {kind: "registered", username: <canonical>}

  Attached to c.set("identity", …) for downstream handlers. Two minor variants
  needed:
  - identity (default): applies format + registry check.
  - identityFormatOnly: format check only, skipping registry check — used for
  POST /users because the caller is by definition allowed to be unregistered
  (otherwise no first user can ever register).
  - POST /reset keeps its §15 exemption and gets neither.

  4. Write gate (§4) — src/middleware/requireWriter.ts

  Runs after identity. If identity is null → 403 forbidden. Otherwise pass
  through. This is the reusable middleware Phases 4–7 will mount on write
  endpoints. To "prove it in isolation" (per the brief), I'll exercise it in
  tests via a tiny Hono app wired to a dummy route — I won't add a fake
  production route.

  5. Tests (node:test, node:assert/strict)

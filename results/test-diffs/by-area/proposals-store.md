# proposals-store
> Proposal store internals: id allocation, install, state transitions, queue determinism, resolve-parent.

## Test inventory
- cc-only: 0 files, 0 dedicated tests (store properties are exercised implicitly via HTTP integration paths)
- cc-openspec: 6 files, 96 tests

## cc-openspec store properties and how cc-only covers (or doesn't) them
- Sequential `mintProposalId` produces `p1, p2, p3` and `mintChangeId` produces `ch1, ch2, ch3`: cc-openspec asserts the literal id strings (`store-ids.test.ts`). cc-only does not assert literal `p1`/`ch1` shapes, but the same scheme is observable indirectly — `proposalsIntegration.test.ts` ("reset across the lifecycle") asserts `second.id === first.id` after reset, which requires deterministic 1-based monotonic minting for both proposal and change counters.
- Counters for proposal and change ids are independent and monotonic: cc-openspec asserts independence and 10-deep monotonicity (`store-ids.test.ts`). cc-only: not observably tested — integration tests never interleave-and-inspect both counters.
- `reset` returns both counters to 1-base: cc-openspec asserts directly (`store-ids.test.ts`). cc-only covers the same property via `proposalsIntegration.test.ts` (`reset across the lifecycle: ids restart`).
- `installProposal` projects a `queued` + `concrete` change into the reviewer's queue with a lowercased key: cc-openspec asserts directly (`store-install.test.ts`). cc-only covers this indirectly via `queueRead.test.ts` ("queued rename entry shape", "routed changes appear only in their reviewer's queue").
- `latent` changes are NEVER projected into any queue: cc-openspec asserts directly (`store-install.test.ts`). cc-only covers via `queueRead.test.ts` ("latent changes are never in any queue") and `proposalsIntegration.test.ts` (Appendix A.1).
- `deferred`-reviewer changes are NEVER in any queue: cc-openspec asserts directly (`store-install.test.ts`). cc-only covers indirectly through latent-change tests (deferred always co-occurs with latent in cc-only's HTTP paths), but cc-only never isolates the `deferred` reviewer kind from `latent` disposition — `queueRead.test.ts` only.
- Queue key is lowercased while `reviewer.username` casing is preserved on the stored change record: cc-openspec asserts directly (`store-install.test.ts`, `store-queue-determinism.test.ts`). cc-only: not observably tested — case-insensitive lookup of the reviewer's *own* queue is implicit only insofar as `queueRead.test.ts` uses lowercased registration; preservation of original casing on the record is not asserted.
- Queue lookup is case-insensitive (`alice`/`ALICE`/`BoB` all resolve): cc-openspec asserts directly (`store-install.test.ts`, `store-queue-determinism.test.ts`). cc-only: not observably tested at the queue layer (`proposalsList.test.ts` covers a *proposer* filter case-insensitivity but that's the list-filter axis, not the per-reviewer queue lookup).
- `changesByProposal` preserves input order; envelope/change retrievable by id: cc-openspec asserts directly (`store-install.test.ts`). cc-only: not observably tested — no integration path inspects per-proposal change order independent of queue ordering.
- `installProposal` throws on duplicate `changeId` within input, duplicate `changeId` already in store, duplicate `proposalId`: cc-openspec asserts directly (`store-install.test.ts`). cc-only: not observably tested — the HTTP submit path cannot construct collisions because ids are minted by the store itself.
- `installProposal` is synchronous (no `await` between mutations): cc-openspec asserts this as a source-grep invariant (`store-install.test.ts`). cc-only: not observably tested.
- `transitionToAccepted` (queued → accepted) clears from queue, records `acceptedBy` verbatim, optionally records `mintedTaxonId`: cc-openspec asserts directly (`store-transitions.test.ts`). cc-only covers state effects via `reviewIntegration.test.ts` ("Appendix A.1 full walkthrough", "rename of a shared taxon propagates ...") — accepted changes leave the queue and effects materialize. Verbatim casing of `acceptedBy` is not separately asserted.
- `transitionToAccepted` throws from latent/accepted/rejected/invalid/unknown sources: cc-openspec asserts directly (`store-transitions.test.ts`). cc-only covers some of these via HTTP error paths in `reviewIntegration.test.ts` (e.g. double-accept rejected), but not exhaustively per source state and not at unit granularity.
- `transitionToRejected` (queued → rejected) records reason (including empty string), removes from queue; throws from non-queued sources: cc-openspec asserts directly (`store-transitions.test.ts`). cc-only covers the happy path via `reviewIntegration.test.ts` (Appendix A.2 reject); empty-string reason and exhaustive forbidden-source enumeration are not observably tested.
- `transitionToInvalid`: queued source stays in queue (broadened invariant), latent source not added to queue; throws on accepted/rejected/invalid/unknown: cc-openspec asserts directly (`store-transitions.test.ts`). cc-only covers the "stays in queue as invalid" property via `phase7Integration.test.ts` (case-3 invalidation flow where Erin sees `state: "invalid"` in her queue). The latent-source path is not isolated.
- `transitionToInvalid` has NO idempotency (second invalid throws): cc-openspec asserts directly (`store-transitions.test.ts`, `store-phase7.test.ts`). cc-only: not observably tested.
- `promoteLatentToQueued`: latent → queued sets concrete reviewer and inserts into queue; promotion overwrites a concrete reviewer; throws from queued/accepted/rejected/invalid/unknown: cc-openspec asserts directly (`store-transitions.test.ts`). cc-only covers latent → queued routing via `reviewIntegration.test.ts` ("rename of a shared taxon propagates") and `phase7Integration.test.ts` ("§14 ownership reassignment reroutes promotion-time reviewer"). Reviewer-overwrite degenerate case and exhaustive forbidden-source enumeration are not observably tested.
- `listQueueFor` returns mixed `{queued, invalid}` entries in ascending `changeId` order, deterministic across reads: cc-openspec asserts directly (`store-transitions.test.ts`, `store-queue-determinism.test.ts`). cc-only covers the ascending-`changeId` order property via `queueRead.test.ts` ("entries ordered by change-id (submission order)") and the mixed `{queued, invalid}` membership via `phase7Integration.test.ts` (Appendix A.2 case-3 invalidation, where the invalid entry remains in Erin's queue alongside other entries until dismiss). Two-reads-same-order determinism not separately asserted.
- `listProposals` filter semantics — no filters / empty filter returns all, `proposer` case-insensitive, `targetRootId` exact-string, intersection, empty-array on no match: cc-openspec asserts directly (`store-queue-determinism.test.ts`). cc-only covers the proposer case-insensitive and unregistered-proposer-empty cases via `proposalsList.test.ts`; `targetRootId` filter and exact-string semantics for it are NOT observably tested in cc-only.
- `dismissInvalidQueued` removes an invalid-in-queue change from the queue; disposition stays `invalid`: cc-openspec asserts directly (`store-phase7.test.ts`). cc-only covers via `phase7Integration.test.ts` ("Appendix A.2 ... with case-3 invalidation and dismiss" — Erin dismisses, entry leaves her queue).
- `dismissInvalidQueued` throws on `queued`/latent-invalid/`accepted`/unknown/already-dismissed: cc-openspec asserts directly (`store-phase7.test.ts`). cc-only: not observably tested — no integration path drives dismiss against forbidden source states or twice on the same id.
- `listQueuedChangesGlobal` returns queued + invalid-in-queue across users in ascending `changeId` order; excludes auto-dismissed, latent-invalid, accepted; empty after reset; includes promoted: cc-openspec asserts directly (`store-phase7.test.ts`). cc-only: there is no `GET /queue/all`-style global queue read in cc-only's test surface, so this is not observably tested.
- Queue-membership relaxation — dismissed invalid + concrete reviewer no longer in queue; invalid-from-queued still in queue until dismissed: cc-openspec asserts directly (`store-phase7.test.ts`). cc-only covers via `phase7Integration.test.ts` (Appendix A.2 dismiss removes from queue; case-3 invalid remains until dismiss).
- `resolveLiveParentId` — `top` ref returns proposal's `topTaxonId`; unknown proposal returns `null`; `live` ref returns id verbatim; `to-create` returns parent's `mintedTaxonId` only when parent is accepted, else `null` (queued/rejected/unknown): cc-openspec asserts directly (`store-resolve-parent.test.ts`). cc-only: not observably tested as an isolated resolver — the property is exercised end-to-end in cascade flows (`cascadeHappy.test.ts`, `phase7Integration.test.ts` Appendix A.2 graft+inner create), but `null`-returning corner cases (queued/rejected/unknown parent) are not directly observable through HTTP.
- Store purity (no HTTP/registry/taxa imports, resolver does not import taxa or `withWriteLock`): cc-openspec asserts as source-grep invariants (`store-queue-determinism.test.ts`, `store-resolve-parent.test.ts`). cc-only: not tested (no equivalent layering invariant).

## cc-only-only scenarios
- None observed. The HTTP integration tests in cc-only exercise compositions of store ops but do not add a store-level property cc-openspec lacks.

## Divergent expectations
- None observed. Both projects agree on id format (`p<n>`, `ch<n>`), 1-base monotonic counters, FIFO-by-`changeId`-ascending queue order, lowercased reviewer queue keys, latent/deferred not in queues, accepted/rejected leave queue, invalid-from-queued stays in queue until dismissed, latent-invalid not in queue, and `to-create` parent resolves to `mintedTaxonId` only after acceptance.

## Shared coverage
- Id counter restart on reset (cc-openspec `store-ids.test.ts`; cc-only `proposalsIntegration.test.ts`).
- Latent and deferred-reviewer changes do not appear in any queue (cc-openspec `store-install.test.ts`; cc-only `queueRead.test.ts`, `proposalsIntegration.test.ts`).
- Queue ordering is ascending by numeric `changeId` (cc-openspec `store-queue-determinism.test.ts`; cc-only `queueRead.test.ts`).
- `queued → accepted` removes the change from the reviewer's queue (cc-openspec `store-transitions.test.ts`; cc-only `reviewIntegration.test.ts`).
- `queued → rejected` removes the change from the reviewer's queue (cc-openspec `store-transitions.test.ts`; cc-only `reviewIntegration.test.ts`).
- `queued → invalid` keeps the change in the reviewer's queue with `state: "invalid"` until dismissed (cc-openspec `store-transitions.test.ts`, `store-phase7.test.ts`; cc-only `phase7Integration.test.ts`).
- `latent → queued` promotion routes to the (re-resolved) concrete reviewer's queue (cc-openspec `store-transitions.test.ts`; cc-only `reviewIntegration.test.ts`, `phase7Integration.test.ts`).
- Dismiss of an invalid-in-queue change removes it from the queue while disposition stays invalid (cc-openspec `store-phase7.test.ts`; cc-only `phase7Integration.test.ts`).
- `to-create` parent resolution after acceptance (mintedTaxonId substitution in cascade flows) (cc-openspec `store-resolve-parent.test.ts`; cc-only `phase7Integration.test.ts` Appendix A.2 graft + inner create).
- Proposer filter on `GET /proposals` is case-insensitive (cc-openspec `store-queue-determinism.test.ts`; cc-only `proposalsList.test.ts`).
